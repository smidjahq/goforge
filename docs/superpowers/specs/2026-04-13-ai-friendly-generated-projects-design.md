# Design: AI-Friendly Generated Projects

**Date:** 2026-04-13
**Status:** Approved

---

## Goal

Every project generated by `goforge create` should be immediately usable by agentic coding tools (Claude Code, OpenAI Codex, Cursor, Gemini CLI, etc.) without the user needing to write any guidance files themselves. It should also include a stack-aware `README.md` so the project is ready to share from day one.

---

## Scope

Two new template files added to the `base/` layer:

- `internal/generator/templates/base/AGENTS.md.tmpl`
- `internal/generator/templates/base/README.md.tmpl`

No changes to the generator engine, config, validator, or TUI. These files are always generated (not gated by an extra flag) because every project benefits from them.

---

## File Placement & Layer Strategy

Both files live in `base/` and use `text/template` conditionals (`{{if eq .Framework "gin"}}` etc.) to emit stack-specific content. This mirrors the existing pattern used by `go.mod.tmpl`, which already conditionally includes framework and DB dependencies from the same base layer.

Physical layer splits (one `AGENTS.md.tmpl` per framework or DB directory) were considered but rejected: these are summary files that describe the whole stack at once, so fragmenting them across layers would either require a Go assembler or duplicate the full skeleton N times.

## `AGENTS.md.tmpl` — Content Structure

The file is rendered with `config.Config`, giving access to `.Name`, `.Module`, `.Framework`, `.DB`, `.Logger`, `.Extras`, `.GoVersion`.

### Sections

**1. Stack summary**
A table listing framework, database, logger, and enabled extras. Gives any AI agent instant orientation.

**2. Commands**
All commands needed to work with the project:
- `go run ./cmd/app` — run the server
- `go build ./cmd/app` — build binary
- `go test ./... -race -count=1` — run tests
- `golangci-lint run ./...` — lint (rendered only when `linter` extra is selected)
- `docker compose up -d` / `docker compose down` — rendered only when `docker` extra is selected
- `migrate -path migrations/ -database "$DATABASE_URL" up/down` — rendered only when `migrations` extra is selected

If the `makefile` extra is selected, a note is added pointing to `make help` as the shortcut for all commands.

**3. Architecture overview**
A text layer diagram showing the Clean Architecture flow:

```
cmd/app → internal/app → controller/http → usecase → repo/persistent
                                                    → repo/webapi
```

One paragraph per layer stating its sole responsibility and what belongs there. This prevents AI agents from placing business logic in handlers or DB code in use cases.

**4. Key conventions**
- Logger interface: always import `pkg/logger`, never a concrete implementation directly
- Entities: domain structs live in `internal/entity/`; no DB tags on domain structs (except GORM, which uses struct tags)
- Config: loaded from `application.yml` + `.env` via Viper and godotenv; add new keys to both files

**5. Extending the project — step-by-step with examples**

*Add a new entity (e.g. `Order`):*
1. Create `internal/entity/order.go` with the domain struct
2. (if `db != none`) Add `internal/repo/persistent/order_repo.go` implementing the repo interface
3. Add `internal/usecase/order.go` with business logic
4. Wire the usecase in `internal/app/app.go`

*Add a new HTTP endpoint:*
1. Add a handler method in `internal/controller/http/handler/`
2. Register the route in `internal/controller/http/router.go` — framework-specific syntax shown inline:
   - Gin: `r.GET("/orders", h.ListOrders)`
   - Chi: `r.Get("/orders", h.ListOrders)`
   - Echo: `e.GET("/orders", h.ListOrders)`
   - Fiber: `app.Get("/orders", h.ListOrders)`

*DB-specific notes* (rendered conditionally per `.DB`):
- **GORM:** Use `AutoMigrate` in `repo.go`; query with `r.db.Model(&Order{}).Where(...).Find(...)`
- **sqlc:** Write SQL in `queries/order.sql`, run `sqlc generate`, implement the generated interface
- **raw:** Follow the `QueryRowContext` / `QueryContext` pattern in `person_repo.go`

*Logger usage* (rendered conditionally per `.Logger`):
- **slog:** `logger.Info("msg", "key", value)`
- **zap:** `logger.Info("msg", zap.String("key", value))`
- **zerolog:** `logger.Info().Str("key", value).Msg("msg")`

## `README.md.tmpl` — Content Structure

### Sections

**1. Title and tagline**
`# {{.Name}}` with a one-line placeholder description the user should fill in.

**2. Stack table**
| Layer | Choice |
|-------|--------|
| Framework | gin / chi / echo / fiber |
| Database | postgres-gorm / ... / none |
| Logger | slog / zap / zerolog |
| Extras | docker, ci, makefile, ... |

**3. Prerequisites**
Always: Go `{{.GoVersion}}+`
Conditionally:
- postgres or mysql: Docker or a running DB instance
- sqlc DB: `sqlc` CLI
- migrations extra: `golang-migrate` CLI
- linter extra: `golangci-lint`

**4. Getting Started**
```bash
cp .env.example .env       # edit DB credentials
docker compose up -d       # (if docker extra selected)
go mod tidy
go run ./cmd/app
```

**5. Project structure**
Annotated directory tree covering `cmd/`, `internal/`, `pkg/`.

**6. Running tests**
```bash
go test ./... -race -count=1
```

**7. Environment variables**
Table of all `.env` keys with short descriptions, rendered from known keys per DB choice.

**8. License**
MIT

## What is NOT changed

- Generator engine (`internal/generator/generator.go`) — no changes needed; `.tmpl` files in `base/` are already walked and rendered automatically
- `config.Config` — no new fields; all needed data is already present
- Validator — no new options; these files are always generated
- TUI prompts — no new prompts
- Tests — new tests added in `internal/generator/` to assert that `AGENTS.md` and `README.md` are present and contain expected stack-specific strings for a representative set of configs

## Test coverage

Add cases to `internal/generator/generator_test.go` (or a new `ai_files_test.go`):
- Assert `AGENTS.md` exists in output dir for a gin+postgres-gorm+zap config
- Assert `README.md` exists and contains the framework name, DB name, logger name
- Assert docker compose commands appear in `AGENTS.md` when `docker` extra is selected
- Assert migrate commands appear when `migrations` extra is selected
- Assert `make help` note appears when `makefile` extra is selected
